Asynchronous JavaScript

JavaScript can only do one thing at a time — yet web apps download data, run timers and react to clicks all at once. This island explains how: the event loop, callbacks, Promises, async/await and the Fetch API.

0 Week at a glance

  • What is asynchronous programming, and how is it different from synchronous code?
  • How does JavaScript — a single-threaded language — handle things that take time?
  • Which tools do we have for async work: callbacks → Promises → async/await (each generation fixes problems of the previous one)?
  • How do we load real data from a server with the Fetch API?

How to study this page: don't just read — press Run on every example and predict the output before it appears. If your prediction is wrong, that's where the learning is.

1 Synchronous vs. asynchronous

Think of a coffee shop with one barista. Synchronous: the barista takes your order, makes your drink, hands it over — and only then serves the next customer. Asynchronous: the barista takes your order, starts the machine, and serves the next customer while your coffee brews.

Synchronous (blocking) code

In synchronous code, every statement must finish completely before the next one starts. The code below always prints in the exact order it is written — nothing can jump the queue:

synchronous.js
console.log("Start");

for (let i = 0; i < 5; i++) {
  console.log("Processing...");
}

console.log("End");

This is simple and predictable — but it has a dark side: while a synchronous task runs, the whole page is frozen. No clicks, no scrolling, no animation. If that loop took 10 seconds, your user would stare at a dead page for 10 seconds.

Asynchronous (event-driven) code

JavaScript in the browser is event-driven: instead of waiting, you register a function to run later, when something happens (a click, a timer, data arriving). Registration is instant — the rest of your code keeps going:

event-driven.html
<button id="btn">Click me</button>
<script>
  const button = document.getElementById("btn");

  button.addEventListener("click", function () {
    alert("Button was clicked!");   // runs LATER, only when clicked
  });

  console.log("Page loaded");        // runs IMMEDIATELY
</script>

"Page loaded" prints right away, even though the click handler was written first. The handler waits patiently, possibly forever, until a click actually happens.

Why we need asynchronous programming

Real web apps constantly do things that take unpredictable amounts of time:

  • Communicating with servers — fetching JSON, submitting forms, loading images
  • Timers — countdowns, auto-save, animations, polling
  • User events — clicks, keystrokes, scrolling
  • Heavy processing — work you don't want to freeze the page

Doing these asynchronously gives us:

BenefitWhat it means
Better user experienceThe page stays responsive — buttons and scrolling keep working while data loads.
Non-blocking I/OSlow operations (network, files) run "in the background" instead of freezing the thread.
Better performanceMultiple slow operations can be in flight at the same time instead of queuing up.
ScalabilityThe same model lets servers (Node.js) handle thousands of simultaneous connections.

2 How JavaScript executes

Here's the puzzle: JavaScript has one thread — one call stack, one thing at a time. So who is counting down the timer while your code keeps running? Answer: the browser.

The mental model has four parts:

  • Call stack — where JavaScript runs your functions, one at a time.
  • Web APIs — browser helpers (timers, network, events) that work outside the JS thread.
  • Callback queue — finished background work drops its callback function here to wait.
  • Event loop — the doorman: when the call stack is empty, it moves the next callback from the queue onto the stack.

Watch it happen — step by step

Press Play and watch how the classic three-line program flows through the system:

event-loop-demo.js
console.log("Start");setTimeout(() => {  console.log("Processing...");}, 2000);console.log("End");
Press Play to start the simulation.

🌐 Web APIs (browser)
⏳ Callback queue
🖥️ Console

Definition — Event loop: the mechanism that lets single-threaded JavaScript appear to do many things at once, by handing slow work to the browser and running the callbacks later, when the stack is free.

Exam favourite: a callback never interrupts running code. Even setTimeout(fn, 0) waits until the current code finishes — the delay is a minimum, not a guarantee.

3 setTimeout & timers

setTimeout is the simplest asynchronous tool: "run this function after a delay". It is our lab rat for the whole week — every async concept can be demonstrated with it.

syntax
setTimeout(callbackFunction, delay, param1, param2, ...);
// callbackFunction : the code to run after the delay
// delay            : time to wait, in milliseconds (1000 ms = 1 second)
// params           : optional arguments passed to the callback

The most important example of the week

Before you press Run — predict the order of the three messages:

timers.js
console.log("Before setTimeout");

const secondInMilliseconds = 1000;
setTimeout(() => {
  console.log("A second has passed");
}, secondInMilliseconds);

console.log("After setTimeout");

"After setTimeout" prints before "A second has passed". JavaScript does not stop at setTimeout — it hands the timer to the browser and keeps going. One second later, the event loop brings the callback back. This one behaviour explains almost everything else this week.

4 Callbacks — the original technique

Generation one of async JavaScript (and still everywhere: event listeners, timers, array methods).

Definition — Callback: a function that you pass as an argument to another function, so that it can be called back later — usually when some work has finished.

This works because in JavaScript, functions are values — you can store them in variables and hand them to other functions like any other data. The receiving function decides when to call them:

callback-basics.js
const doSomething = (cb) => {
  console.log("Doing something...");
  cb();                      // "calling back" the function we were given
};

const nextStep = () => {
  console.log("Callback called");
};

doSomething(nextStep);        // pass the FUNCTION itself — no parentheses!

Classic mistake: writing doSomething(nextStep()) — with parentheses — calls nextStep immediately and passes its return value. Pass the function name only: doSomething(nextStep).

Callbacks for async work

Callbacks shine when the work takes time. Here fetchData simulates a slow server request (2 seconds) and delivers the result through the callback — that's how async functions "return" values when the caller has already moved on:

async-callback.js
function fetchData(callback) {
  console.log("Fetching data...");
  setTimeout(() => {
    const data = { name: "Hala", role: "Professor" };
    callback(data);              // deliver the result via the callback
  }, 2000);
}

function processData(data) {
  console.log("Data received:", data);
}

fetchData(processData);          // pass processData as the callback

5 Callback hell

Callbacks fall apart when async steps depend on each other: get the user → then get their posts → then get the comments. Each step must nest inside the previous callback, and the code slides diagonally off the screen:

fetchUser(1, (user) => { fetchPosts(user, (posts) => { fetchComments(posts, (comments) => { console.log("Final data:", comments); // 3 levels deep already… }); }); });

This shape is called the pyramid of doom. Try the full version:

callback-hell.js
function fetchUser(userId, callback) {
  setTimeout(() => {
    console.log("Fetched user");
    callback({ id: userId, name: "Dinn" });
  }, 1000);
}

function fetchPosts(user, callback) {
  setTimeout(() => {
    console.log("Fetched posts for " + user.name);
    callback(["Post 1", "Post 2"]);
  }, 1000);
}

function fetchComments(posts, callback) {
  setTimeout(() => {
    console.log("Fetched comments for " + posts[0]);
    callback(["Comment 1", "Comment 2"]);
  }, 1000);
}

// Each step nests inside the previous one…
fetchUser(1, (user) => {
  fetchPosts(user, (posts) => {
    fetchComments(posts, (comments) => {
      console.log("Final data:", comments);
    });
  });
});

Why this hurts in real projects:

  • Readability — the logic reads diagonally instead of top-to-bottom.
  • Error handling — every level needs its own manual error check; there is no single place to catch failures.
  • Maintenance — inserting or reordering a step means re-nesting everything below it.

JavaScript's answer, added in ES6 (2015): Promises.

6 Promises

A Promise is exactly what it sounds like: an IOU. "I don't have your value yet, but I promise you'll either get it — or a reason why not."

Definition — Promise: an object that represents a value that will be available in the future. It gives all asynchronous code one standard interface.

The three states

A promise starts pending and settles exactly once, into one of two final states. Try it yourself — you play the role of the async operation:

⏳ pending
The promise is pending — its .then() and .catch() handlers are waiting…
  • Pending — initial state; the operation hasn't finished.
  • Fulfilled (resolved) — the operation succeeded; the promise now holds a value.
  • Rejected — the operation failed; the promise now holds a reason (an error).

Notice in the demo: once settled, the buttons stop working. A promise can only settle once — calling resolve or reject again does nothing.

Creating a promise

You build one with new Promise(executor). The executor function receives two functions from JavaScript itself: call resolve(value) on success, reject(reason) on failure:

creating-a-promise.js
const myPromise = new Promise((resolve, reject) => {
  let success = true;              // flip to false to see the rejection path

  setTimeout(() => {
    if (success) {
      resolve("Promise resolved successfully!");
    } else {
      reject("Promise rejected! Something went wrong.");
    }
  }, 2000);                        // simulate a slow operation
});

myPromise
  .then((message) => console.log("SUCCESS:", message))
  .catch((error) => console.log("FAILURE:", error));

Consuming a promise with .then()

.then(callback) registers what should happen when the promise fulfills. The callback receives the resolved value as its argument. Registering is instant — the callback runs later:

consuming.js
const setTimeoutPromise = (time) => {
  return new Promise((resolve, reject) => {
    setTimeout(() => {
      resolve();
    }, time);
  });
};

console.log("Before setTimeoutPromise");

setTimeoutPromise(1000).then(() => console.log("one second later"));

console.log("After setTimeoutPromise");

Same pattern as section 3 — "After" prints before "one second later" — but now the timer is wrapped in a promise, which we can chain, combine, and await. For quick tests there are also shortcuts: Promise.resolve(value) creates an already-fulfilled promise, and Promise.reject(reason) an already-rejected one (which is an Uncaught (in promise) error if you never attach a .catch).

Chaining promises

The superpower: .then() returns a new promise, so chains read top-to-bottom. Whatever you return inside a .then becomes the input of the next one:

chaining.js
const fetchData = new Promise((resolve) => {
  setTimeout(() => {
    resolve(10);
  }, 1000);
});

fetchData
  .then((number) => {
    console.log("First then:", number);    // 10
    return number * 2;                     // passed to the next .then()
  })
  .then((newNumber) => {
    console.log("Second then:", newNumber); // 20
    return newNumber + 5;
  })
  .then((finalResult) => {
    console.log("Final result:", finalResult); // 25
  });

Escaping callback hell

Rewrite the section-5 functions to return promises instead of accepting callbacks, and the pyramid flattens into a readable pipeline:

promise-chain.js
function fetchUser(userId) {
  return new Promise((resolve) => {
    setTimeout(() => {
      console.log("Fetched user");
      resolve({ id: userId, name: "Dinn" });
    }, 1000);
  });
}

function fetchPosts(user) {
  return new Promise((resolve) => {
    setTimeout(() => {
      console.log("Fetched posts for " + user.name);
      resolve(["Post 1", "Post 2"]);
    }, 1000);
  });
}

function fetchComments(posts) {
  return new Promise((resolve) => {
    setTimeout(() => {
      console.log("Fetched comments for " + posts[0]);
      resolve(["Comment 1", "Comment 2"]);
    }, 1000);
  });
}

// Flat, readable, top-to-bottom:
fetchUser(1)
  .then(fetchPosts)
  .then(fetchComments)
  .then((comments) => {
    console.log("Final data:", comments);
  });

Handling errors: .then / .catch / .finally

MethodRuns when…
.then(cb)the promise is fulfilled (and to continue the chain).
.catch(cb)the promise — or any earlier step in the chain — is rejected or throws.
.finally(cb)always, success or failure — perfect for cleanup like hiding a spinner.
catch-finally.js
function getData() {
  return new Promise((resolve, reject) => {
    setTimeout(() => {
      // Flip Math.random() < 0.5 for a coin-toss between success and failure
      if (Math.random() < 0.5) {
        resolve("fresh data");
      } else {
        reject(new Error("server timeout"));
      }
    }, 800);
  });
}

getData()
  .then((data) => console.log("Step 1 complete:", data))
  .then(() => console.log("Step 2 complete"))
  .catch((error) => console.log("Something went wrong:", error.message))
  .finally(() => console.log("All done! (runs either way)"));

Run this a few times — sometimes it succeeds, sometimes it fails. Notice that one single .catch at the end guards the entire chain, and .finally runs every time.

Callbacks vs. Promises

FeatureCallbacksPromises
DefinitionA function passed as an argument, executed later.An object representing a future value or error.
SyntaxFunctions passed as parameters..then() and .catch().
ReadabilityCan lead to callback hell (deep nesting).Flat chains, reads top-to-bottom.
Error handlingManual, at every level.Built-in .catch() for the whole chain.
ChainingDifficult; requires nesting.Natural with .then().
FlexibilityPoor for complex flows.Composable (Promise.all, etc.).

Promises aren't perfect either: long chains can still get hard to debug, error handling across chains takes care, and forgotten handlers can leak. That's why ES2017 added one more layer of comfort…

7 Async / await — the modern way

async/await (ES2017) is syntax sugar built 100% on top of Promises. Nothing new happens under the hood — your async code just looks synchronous.

  • async before a function makes it always return a Promise.
  • await pauses that function until a promise settles, then hands you the resolved value.

async functions return promises

These two functions are exactly equivalent — returning a plain value from an async function wraps it in a resolved promise automatically:

Returning a value
a.js
async function f() {
  return 1;
}

f().then(console.log); // 1
Returning a promise
b.js
async function f() {
  return Promise.resolve(1);
}

f().then(console.log); // 1

Rules of the road

  • You can only use await inside an async function (or at a module's top level).
  • await works on promises: fulfilled → returns the value; rejected → throws the error.
  • While an async function is paused, the browser is NOT blocked — other JavaScript keeps running. Only that one function waits.
  • await makes steps run one after another — great for dependent steps, wasteful for independent ones.
breakfast.js
function makeCoffee() {
  return new Promise((resolve) => {
    setTimeout(() => {
      resolve("☕ Coffee is ready!");
    }, 2000);                       // simulate a 2-second brew
  });
}

async function prepareBreakfast() {
  console.log("Preparing pancakes...");
  const coffee = await makeCoffee();   // pause HERE until the promise resolves
  console.log(coffee);
  console.log("🥞 Breakfast is ready!");
}

prepareBreakfast();
console.log("(meanwhile, the rest of the program keeps running…)");

Note the last line: it prints before the coffee is ready, proving that await paused only prepareBreakfast, not the whole program.

The same logic, two styles

Here is the callback-hell scenario one final time — promise chain on the left, async/await on the right:

Promise chain
then-style.js
function loadData() {
  fetchUser(1)
    .then(function (user) {
      return fetchPosts(user.id);
    })
    .then(function (posts) {
      return fetchComments(posts[0].id);
    })
    .then(function (comments) {
      console.log(comments);
    })
    .catch(function (error) {
      console.log(error);
    });
}
Async / await
await-style.js
async function loadData() {
  try {
    let user     = await fetchUser(1);
    let posts    = await fetchPosts(user.id);
    let comments = await fetchComments(posts[0].id);
    console.log(comments);
  } catch (error) {
    console.log(error);
  }
}

Same behaviour, but the right-hand version reads like a simple recipe: get the user, get the posts, get the comments.

Handling rejections with try / catch

Because a rejected promise throws at the await, async functions use the classic try…catch statement:

try-catch.js
function riskyOperation() {
  return new Promise((resolve, reject) => {
    setTimeout(() => reject(new Error("the server is on fire 🔥")), 700);
  });
}

async function main() {
  try {
    console.log("Attempting operation…");
    const result = await riskyOperation();   // this THROWS when rejected
    console.log("Success:", result);         // skipped
  } catch (error) {
    console.log("Caught it:", error.message);
  }
}

main();

The golden rule: every await that can reject must be covered by a try/catch (or the returned promise must get a .catch). Uncovered rejections crash into the console as Uncaught (in promise).

8 The Fetch API

Everything so far used fake timers. Time for the real thing: fetch() is the browser's built-in API for reading content from a URL — and it returns a Promise.

syntax
fetch(url[, options])   // returns a Promise that resolves to a Response object

The Response object you get is not the data itself — it's the envelope:

MemberWhat it gives you
response.statusThe HTTP status code (200 OK, 404 Not Found, 500 Server Error…).
response.oktrue if the status is 200–299 — always check this!
response.text()Reads the body as a plain string — returns another Promise.
response.json()Reads the body and parses it as JSON into a JavaScript object — also a Promise.

Two awaits, not one: reading the body is a second asynchronous step. That's why every fetch has two stages: await fetch(url) then await response.json().

Fetch with .then() — live demo

This demo loads albums.json — a real file sitting next to this page on the server. Watch the two-stage pattern: parse the response, then use the data:

fetch-then.js · live network request
fetch("albums.json")
  .then((response) => response.json())   // parse the body as JSON
  .then((data) => {
    console.log("Loaded", data.length, "albums");
    console.log("First album:", data[0].title);
  })
  .catch((error) => console.error("Error:", error.message));

The robust version — always check response.ok

fetch only rejects on network failure. A 404 page still "succeeds" — with a useless body! Production code checks the status before parsing. This example calls a real public test API, JSONPlaceholder:

fetch-robust.js · live network request
const apiUrl = "https://jsonplaceholder.typicode.com/posts";

fetch(apiUrl)
  .then((response) => {
    if (!response.ok) {
      throw new Error("Network response was not ok: " + response.status);
    }
    return response.json();
  })
  .then((data) => {
    console.log("Received", data.length, "posts");
    console.log("First post title:", data[0].title);
  })
  .catch((error) => {
    console.error("There was a problem with the fetch operation:", error.message);
  });

Fetch with async/await — a step-by-step walkthrough

load-json.js · live network request
async function loadJson() {
  const response = await fetch("albums.json");
  const json = await response.json();
  console.log("Got " + json.length + " albums:", json.map(a => a.title).join(", "));
}

loadJson();
console.log("after loadJson");

Here is exactly what the JavaScript engine does, in order:

  1. loadJson() is called and starts running.
  2. At the first await: fetch("albums.json") starts, and loadJson is paused. The engine returns out of the function and continues below — printing "after loadJson".
  3. The response arrives; the engine resumes loadJson. The await expression evaluates to the resolved value — the response object.
  4. At the second await: response.json() starts, and the function pauses again until parsing completes.
  5. Finally, json holds a real JavaScript array, and the last console.log runs.

That is why the output shows "after loadJson" first, even though it's the last line of the file — the exact event-loop behaviour from section 2, now with real network requests.

9 Choosing a technique

Three generations, one family — async/await is built on Promises, which tamed callbacks.

Callbacks (old)Promises (ES6 · 2015)Async/await (ES2017)
StyleFunction passed in, nested.then().catch() chainLooks synchronous
Error handlingManual at each levelOne .catch() per chainStandard try/catch
Dependent stepsPyramid of doomFlat chainOne line per step
DebuggingHardMediumEasiest — normal stack traces
Still used forEvent listeners, timers, array methodsLibraries, combining tasks (Promise.all)Nearly all new application code

Rule of thumb for your projects: write new async code with async/await + try/catch; use .then() when it's genuinely shorter; and remember that event listeners (addEventListener) are — and will always be — callbacks.

10 Key takeaways

  1. JavaScript is single-threaded — the event loop + browser Web APIs create the illusion of doing many things at once.
  2. setTimeout never blocks: code after it runs first, the callback runs when the timer fires and the stack is empty.
  3. A callback is a function passed to another function to be called later. Nesting dependent callbacks creates callback hell.
  4. A Promise is a future value with three states: pending → fulfilled / rejected. It settles exactly once.
  5. Consume promises with .then(), handle failures with .catch(), clean up with .finally() — and chain by returning values.
  6. async/await is sugar over promises: async functions return promises; await pauses the function (not the browser!) and throws on rejection — so wrap it in try/catch.
  7. fetch(url) returns a promise for a Response; check response.ok, then parse with response.json() — a second await.

11 Self-check quiz

No grades, no pressure — instant feedback. Can you get all six?

Q1. What does this print, in order?
console.log("A"); setTimeout(() => console.log("B"), 0); console.log("C");

Even with a 0 ms delay, the callback goes through the Web API → callback queue → event loop journey, and the event loop only delivers it once the call stack is empty. So: A, C, B.

Q2. Which are the three states of a Promise?

A promise starts pending, and settles exactly once into fulfilled (resolved) or rejected.

Q3. What does an async function always return?

async function f() { return 1; } returns a promise that resolves to 1 — that's why you can call f().then(...).

Q4. While an async function is paused at an await, the browser is…

Only that one function is suspended. The main thread is released — that's the whole point of async programming.

Q5. Why do we call response.json() after fetch()?

fetch resolves with a Response envelope. Reading + parsing the body is a second asynchronous operation, which is why it needs its own await or .then().

Q6. A promise rejects inside an async function with no try/catch. What happens?

The golden rule: every await that can reject must be covered by try/catch (or a .catch on the returned promise). Otherwise the rejection surfaces as an uncaught error.
